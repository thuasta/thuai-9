## 🔴 Critical

### 1. Evaluator crashes on startup: `Index` used in SubmissionMatchLog but never imported (NameError)
- **Where:** `thuai-9-server/evaluator/main.py` 23,81-83 · _eval-runner-infra_, conf high
- **Why:** Verified via AST scan: `Index imported? False`. The name is referenced at module level inside the class body, so the import error fires unconditionally at process start. Git blame shows commit a39fb82 added both the `Index(...)` usage AND `Index` to the import, but the merge commit 0ef8aa8 (resolving a conflict with the branch that added `Boolean`) produced an import line containing `Boolean` while dropping `Index`. The usage at line 82 was kept. Result: the evaluator container enters a crash-loop (restart: unless-stopped) and never compiles a single submission or runs a single match — the whole evaluation platform is dead.
- **Trigger:** docker compose up evaluator (or run `python main.py`). Process dies immediately with `NameError: name 'Index' is not defined` raised while importing main.py at the `class SubmissionMatchLog` definition.
- **Fix:** Add `Index` to the import on line 23: `from sqlalchemy import Boolean, Column, DateTime, ForeignKey, Index, Integer, String, Text, func`.

### 2. Per-tick broadcast reads order book / pending orders without the TradingDay lock, racing socket-thread order handlers (crash + game freeze)
- **Where:** `server/src/thuai/Program.cs` 239-298 · _server-connection-protocol_, conf high
- **Why:** SortedSet<Order>, List<Order>, and Dictionary are not thread-safe; enumerating one on the tick thread while another thread Add/Removes inside the lock throws InvalidOperationException: 'Collection was modified; enumeration operation may not execute' (or returns corrupted data). The lock on the write side gives no protection because the read side never acquires it. The thrown exception propagates out of BroadcastGameState -> Game.Tick() -> the GameController tick loop's Task.Run, which has NO try/catch (GameController.cs:34-47). The faulted task ends the `while (IsRunning)` loop, so all ticks stop and the match freezes permanently.
- **Trigger:** Run a match with >=1 connected agent. Each tick the server broadcasts the order book. Have the agent send LIMIT_BUY/LIMIT_SELL at a steady rate (e.g. every few ms). Eventually a place/match Add/Remove on _bids/_asks/_orders/_pendingOrders coincides with the broadcast's enumeration of that collection -> InvalidOperationException -> tick loop task faults -> game hangs (IsRunning loop exits, no more ticks, clients stop receiving GAME_STATE).
- **Fix:** Acquire TradingDay._lock around the market/player-state reads (or expose lock-guarded snapshot methods on TradingDay/OrderBook/MatchEngine that copy under the lock), and snapshot game state under Game._lock before broadcasting. At minimum wrap the AfterGameTickEvent invocation / tick body in try/catch so a broadcast exception cannot kill the tick loop.

### 3. Integer overflow on price*quantity lets a buy order create unlimited Mora / NAV
- **Where:** `server/src/thuai/GameLogic/MatchEngine.cs` 28-63 · _server-matchengine_, conf high
- **Why:** I added a probe test (MatchEngine, fee rate 0.0002, fresh player with 1,000,000 Mora) and submitted a BUY at price=9_000_000_000, quantity=2_000_000_000. The unchecked product wrapped to -446744073709551616. Output: order accepted=True, A.Mora=446833422525293526, A.FrozenMora=-446833422524293526 (i.e. ~4.46e17 Mora created from thin air in the available balance). I then exploited it: the player bought 1000 real gold from another player at price 1 (now affordable because available Mora is astronomically inflated), and the player's NAV at mid-price 1000 rose from 2,000,000 to 2,999,000 — ~1,000,000 of value created out of nothing, directly inflating the settlement score (CalculateNAV uses Mora + FrozenMora + Gold*midPrice, Player.cs:124). The corrupted order also rests in the book; if an opponent ever matches it, ExecuteTrade recomputes `price*quantity` (line 206) and overflows again.
- **Trigger:** Connect as a player and send {messageType:'LIMIT_BUY', token:..., price:9000000000, quantity:2000000000}. SubmitOrder accepts it; the player's available Mora becomes ~4.46e17 and FrozenMora ~-4.46e17. The player can then place real orders / buy real gold using the inflated Mora, breaking money conservation and inflating NAV at settlement.
- **Fix:** Reject out-of-range orders before any arithmetic: enforce sane upper bounds on price and quantity (e.g. price and quantity each <= a configured max such that max_price*max_quantity*(1+maxFeeRate) < long.MaxValue and is also <= InitialMora), and/or use `checked { ... }` / Int128 / overflow detection around `price*quantity` in SubmitOrder (line 43) and ExecuteTrade (line 206) and reject on overflow.


## 🟠 High

### 4. Untrusted contestant agent containers join the internal network (postgres, submission-api) with internet egress and no isolation hardening
- **Where:** `thuai-9-server/evaluator/runner.py` 33-49 · _eval-runner-infra_, conf high
- **Why:** This is untrusted code by definition (it is the contestant submission). On the shared internal network it can: directly attack/port-scan postgres and brute the DB password, hit submission-api on :8000 bypassing all nginx rate-limits and the WAF-style /api/register and /api/upload throttles, and reach the live server out-of-band. With internet egress it can exfiltrate, mine, or launch DDoS. With no pids_limit a fork bomb evades the 256m cap. Confirmed: postgres is on `networks: [internal]` (compose lines 11-13) and agents join the prefixed `thuai-9-server_internal` (config.py:13).
- **Trigger:** Submit a Dockerfile whose entrypoint runs e.g. `nc thuai9-postgres 5432` / `curl http://submission-api:8000/...` / `:(){ :|:& };:`. During its match the agent container reaches postgres + submission-api on the internal net and the open internet, and a fork bomb spawns unbounded processes within the (un-pids-limited) container.
- **Fix:** Spawn agents on a dedicated, isolated network created with `internal: true` (no internet) that contains ONLY the live game server, not postgres/submission-api. Add `pids_limit`, `cap_drop=['ALL']`, `security_opt=['no-new-privileges']`, `read_only=True` (with a small tmpfs), and force a non-root user. Do not place untrusted agents on the same bridge as the datastore and API.

### 5. Untrusted Dockerfile build runs with no wall-clock timeout — a single slow/infinite build hangs the global compile loop
- **Where:** `thuai-9-server/evaluator/compiler.py` 119-153 · _eval-runner-infra_, conf high
- **Why:** A malicious or accidental Dockerfile (`RUN sleep infinity`, `RUN while true; do :; done`, a huge `FROM`/download, or an OOM/forkbomb build step) blocks the build generator indefinitely. Because compile_loop is strictly serial and picks one pending row at a time, every other team's submission stays stuck in `pending`/`compiling` forever — a trivial denial of service against the whole compile queue. Confirmed: grep for `network|buildarg|cpu|mem|nano|pids|ulimit|timeout` in compiler.py returns NONE, so the build also has no network isolation or resource caps.
- **Trigger:** Upload a zip with `Dockerfile` containing `FROM alpine` + `RUN sleep 100000`. compile_loop blocks on it; all subsequent submissions never compile.
- **Fix:** Run the build with a bounded timeout (e.g. wrap the build-log iteration in a deadline and abort/cancel the build, or use BuildKit with `--timeout`), and set `network_mode='none'` plus CPU/memory build constraints. Consider compiling in a worker with a hard kill so one bad build cannot wedge the queue.

### 6. Evaluator mounts /var/run/docker.sock (docker-out-of-docker) = host-root-equivalent, while also running untrusted build/run logic in-process
- **Where:** `thuai-9-server/docker-compose.yml` 54-59 · _eval-runner-infra_, conf high
- **Why:** Docker-socket access is universally equivalent to host root. The evaluator simultaneously (a) extracts and builds untrusted contestant zips/Dockerfiles in-process and (b) holds the docker.sock. The build itself is contestant-controlled and runs on the host daemon with no network/resource isolation (see compiler finding), so a Dockerfile build step or a docker-py deserialization bug becomes a host takeover primitive. There is no socket-proxy (e.g. tecnativa/docker-socket-proxy) restricting which API calls the evaluator may make.
- **Trigger:** Any RCE in the evaluator process (or a build-time escape) lets the attacker call the mounted Docker API: `docker -H unix:///var/run/docker.sock run -v /:/mnt alpine cat /mnt/etc/shadow`, or launch a privileged container, achieving full host compromise.
- **Fix:** Do not give the evaluator raw docker.sock. Front it with a docker-socket-proxy that whitelists only the needed endpoints (images/build, containers/create/start/stop/remove with label filters), or move match orchestration to a rootless/sysbox or gVisor runtime. At minimum keep all untrusted parsing/building in a separate, socket-less worker.

### 7. No per-team submission cap and uploaded zips are never deleted → unbounded disk fill (DoS)
- **Where:** `thuai-9-server/submission-api/app/routers/submissions.py` 122-162 · _eval-submission-api_, conf high
- **Why:** The only os.remove on the upload path is the rollback cleanup at submissions.py:156. grep across submission-api/app shows no other deletion, and evaluator/compiler.py:93 only reads upload_dir/{submission_id}.zip, it never removes it. The upload_dir is the shared submission-data Docker volume (docker-compose.yml mounts ${PWD}/submission-data into both submission-api and evaluator). An authenticated team can therefore POST /submissions/upload in a loop, each call persisting up to 10 MB (MAX_UPLOAD_BYTES) to that shared volume forever, filling the disk and taking down postgres/evaluator that share the host.
- **Trigger:** Authenticate as any team, then repeatedly POST a ~10MB .zip to /api/submissions/upload (nginx upload zone allows 60 req/min). Each request adds one submissions row plus a 10MB file under submission-data/uploads/{id}.zip that is never reclaimed; after a few thousand requests the volume is full.
- **Fix:** Enforce a per-team active-submission cap in upload_submission (e.g. reject when the team already has N un-deleted submissions, or recycle the oldest). Delete upload_dir/{id}.zip after the compiler consumes it (or after status transitions to ready/failed), and add a retention/GC job for orphaned zips.

### 8. Unbounded per-socket send queue: a stalled/non-reading client makes server memory grow without bound
- **Where:** `server/src/thuai/Connection/AgentServer/AgentServer.MessageSending.cs` 38-97 · _server-connection-protocol_, conf high
- **Why:** Producer rate (every tick, unconditional) exceeds a stalled consumer's drain rate with no upper bound on the queue, so the ConcurrentQueue grows unboundedly and the process's managed heap grows until OOM. Unlike the receive path there is no overflow guard. A single malicious or buggy client (even an observer, which only needs to send a HELLO with role=observer to start receiving PublishToAll/PublishToObservers traffic) can exhaust server memory.
- **Trigger:** Connect as observer (send {messageType:'HELLO', role:'observer'}) so the socket gets a role and starts receiving PublishToAll broadcasts, then stop reading from the socket (don't ACK/consume). The server enqueues every tick's GAME_STATE/MARKET_STATE forever; the unbounded queue grows until the server runs out of memory.
- **Fix:** Bound _socketMessageSendingQueue (e.g. drop oldest or disconnect the socket when Count exceeds a cap), mirroring the MaxMessageQueueSize guard used on the receive side; consider coalescing stale state messages (keep only the latest GAME_STATE/MARKET_STATE per socket).

### 9. Session takeover in default config: any socket with a valid token binds as that player (private state + order authority) and evicts the real client
- **Where:** `AgentServer/AgentServer.MessageReceiving.cs:97-114; AgentServer.EventHandler.cs:13-23; AgentServer.MessageSending.cs:49-57; server/config/config.json (acceptAnyToken:true)` · _server-connection-protocol (live-observed)_, conf high
- **Why:** Publish(msg,token) fans out to EVERY socket whose token matches; BindPlayerSocket evicts a prior socket for the same token without closing it. Shipped config.json sets acceptAnyToken:true and the server binds 0.0.0.0, so ANY network client sending a token-bearing message as player1/player2 takes over that session: gets private PLAYER_STATE and can submit orders, kicking the real agent. Commit a39fb82 ('stop exposing live-server tokens') shows tokens have leaked.
- **Trigger:** OBSERVED LIVE: a pre-HELLO probe sending CANCEL_ORDER{token:player1} while the real player1 agent was connected received 49 private PLAYER_STATE messages. (An honest HELLO-observer received 0 — observer isolation is correct.)
- **Fix:** Default acceptAnyToken:false; bind a configurable interface; reject a 2nd bind for an actively-connected token (or require a per-session secret); close the displaced Fleck socket.

### 10. Order book and pending-orders collections are read by the broadcast thread with no lock while a socket thread mutates them
- **Where:** `server/src/thuai/GameLogic/OrderBook.cs` 84-178 · _server-matchengine_, conf high
- **Why:** Game.Tick takes Game._lock (Game.cs:62) but invokes AfterGameTickEvent at line 90 after releasing it; BroadcastGameState therefore reads the book/pending lists unlocked. Player message handlers run on independent socket threads (Task.Run ParseMessageLoop, AgentServer.SocketManagement.cs:20) and mutate the same collections under TradingDay._lock. Two threads, two unrelated locks, shared non-concurrent collections being enumerated and mutated simultaneously. .NET SortedSet/List/Dictionary throw InvalidOperationException ('Collection was modified; enumeration operation may not execute') on concurrent enumerate+mutate, or return torn/corrupt data, crashing the server mid-match.
- **Trigger:** During a TradingDay, while the per-tick BroadcastGameState is iterating _bids/_asks (GetVisibleBids) or _pendingOrders (GetPlayerPendingOrders), a player sends a LIMIT_BUY whose handler runs on the socket thread and calls SubmitOrder (List.Add) or ProcessDay (List.RemoveAll + SortedSet.Add/Remove). The unsynchronized concurrent access throws InvalidOperationException in the broadcast loop (or yields corrupt market data).
- **Fix:** Acquire TradingDay._lock around the order-book/pending-order reads in BroadcastGameState (or take a defensive snapshot of bids/asks/pending under the lock and broadcast from the snapshot), or invoke AfterGameTickEvent while still holding Game._lock and route all order-book mutation and reads through a single lock. Long term, OrderBook/MatchEngine read accessors should lock or return immutable snapshots.

### 11. Self-trades (wash trades) are not prevented and move LastPrice/MidPrice + inflate volume at near-zero cost
- **Where:** `server/src/thuai/GameLogic/MatchEngine.cs` 143-249 · _server-matchengine_, conf high
- **Why:** My probe placed a resting BUY at 1000 for player A, then submitted A's own SELL at 1000. A single trade executed with IsWashTrade=True; TotalVolume jumped from 0 to 10 and LastPrice was set to 1000, while A's only net cost was the two transaction fees (value delta = -4 at the 0.0002 rate). A player who holds gold can post a resting BUY at an inflated price and self-cross it to push LastPrice up; when the opposite side of the book is then empty/thin, MidPrice = LastPrice, raising the player's settlement NAV (Mora + FrozenMora + Gold*midPrice) at essentially the cost of two tiny fees. TotalVolume is also broadcast as public market data (Program.cs:262), so volume can be fabricated.
- **Trigger:** As a player: place a resting limit BUY at a high price, then send a crossing LIMIT_SELL at that price into your own bid. A self-trade fires, LastPrice is set to your chosen price and TotalVolume is inflated; if the book is one-sided at settlement, MidPrice equals that LastPrice and your NAV is inflated.
- **Fix:** In ProcessOrder's match loop, when the best opposite order belongs to the same player, either skip/forbid the self-cross (cancel-on-self per exchange convention) or do not update LastPrice/TotalVolume for self-trades (guard lines 231-232 with `taker.PlayerToken != maker.PlayerToken`). Prefer rejecting the self-cross outright so it cannot influence public market data or settlement price.

### 12. StopLossBlade downside-protection price inflates the permanent settlement NAV that decides final ranking
- **Where:** `server/src/thuai/GameLogic/Player.cs` 114-126 · _server-player-game_, conf high
- **Why:** StopLossBlade is documented as reducing *intra-day* downside loss to 20% (RiskControlCards.cs:7). Immunity expires only when `currentDay > ImmuneUntilDay` (RiskControlCards.cs:28). Activating the blade in the last 1-2 ticks of a trading day (e.g. tick 29 of 30 -> ImmuneUntilDay=31) leaves IsImmune=true at settlement, and CalculateNAV(midPrice) then values the player's gold above mark-to-market. There is no step that clears immunity or marks NAV at the true price before settlement, so the inflated value is baked into CumulativeNavs forever and skews the winner determination.
- **Trigger:** 2-player finite game, TradingDayTicks=30. Player A holds 1000 gold; price drops near day end. A draws+activates StopLossBlade at tick 29 (ProtectedMidPrice = mid at 29). At tick 30 price is lower; trading day finishes. CalculateSettlement() calls A.CalculateNAV(lowerMid): because IsImmune && lowerMid < ProtectedMidPrice, A is credited ~ProtectedMidPrice per gold instead of lowerMid. A's CumulativeNavs (and thus Scoreboard) is inflated above true wealth and can flip the final ranking.
- **Fix:** Compute the permanent settlement NAV at the true mid price (ignore the protected-price cushion), or clear IsImmune/ProtectedMidPrice (or pass a `forSettlement` flag) before CalculateSettlement so the protection only affects the live trading-day display, not the cumulative score that decides the match.

### 13. Recorder/broadcast read order book without lock → concurrent List/Dictionary mutation throws and faults the game loop
- **Where:** `server/src/thuai/Program.cs` 601-611, 269; GameLogic/TradingDay.cs:223-230; GameLogic/MatchEngine.cs:11,62,118; GameLogic/OrderBook.cs:19,90,174 · _server-recorder-runtime_, conf high
- **Why:** A List/Dictionary being enumerated with .Where().ToList() on the recorder thread while another thread calls .Add()/indexer on it throws InvalidOperationException ('Collection was modified; enumeration operation may not execute'). The reader holds no lock and the writer holds only TradingDay._lock, so there is zero mutual exclusion between them. Confirmed: _pendingOrders is 'private readonly List<Order> _pendingOrders = new();' (MatchEngine.cs:11), GetPlayerPendingOrders has no lock body (TradingDay.cs:223-230), and HandleLimitBuy locks TradingDay._lock (TradingDay.cs:144-146).
- **Trigger:** Two agents connected. While a tick is being broadcast/recorded, one agent sends LIMIT_BUY so HandleLimitBuy adds to _pendingOrders on a ParseMessageLoop task at the same moment RecordGameState/BroadcastGameState enumerates that player's pending orders. Under load this throws 'Collection was modified'. Because Game.Tick() invokes the event with no try/catch and the loop is a fire-and-forget Task.Run, the exception faults the task and the server hangs (see related IsRunning finding).
- **Fix:** Take a consistent snapshot of pending orders inside TradingDay._lock (e.g. add a locked GetPlayerPendingOrders, or snapshot all data the recorder/broadcaster needs while still holding Game._lock and pass immutable copies into AfterGameTickEvent). At minimum, make GetPlayerPendingOrders/MatchEngine/OrderBook reads lock the same lock the writers use.

### 14. Game loop runs in a fire-and-forget Task with no exception handling; any unhandled tick exception hangs the whole server
- **Where:** `server/src/thuai/GameController/GameController.cs` 30-48; Program.cs:126-129 · _server-recorder-runtime_, conf high
- **Why:** If Game.Tick() (or any AfterGameTickEvent handler such as the recorder, which can throw the concurrent-collection exception above, or any game-logic exception) throws, the Task faults silently. IsRunning is never set false, so Main's poll loop spins forever. The process never reaches the shutdown path (Program.cs:131-147), so recorder.Flush()/statRecorder.Flush()/SaveResults are never called — the replay tail and final result.json are lost, and the evaluator (which polls result.json until GAME_TIMEOUT_SECONDS) eventually times out the match.
- **Trigger:** Cause any exception inside a tick or an AfterGameTickEvent handler (e.g. trigger the concurrent _pendingOrders modification). The game-loop Task faults, IsRunning stays true, the server process hangs indefinitely and produces no final result.
- **Fix:** Wrap the loop body in try/catch: on exception, Log.Fatal and set IsRunning=false (and surface the fault). Optionally observe the Task and propagate faults to Main so the finally block (Log.CloseAndFlush / recorder flush) still runs.


## 🟡 Medium

### 15. Python SDK silently ignores DAY_SETTLEMENT — Python agents never see monthly settlement results
- **Where:** `sdk-python/sdk_python/agent.py` 234-257 · _Cross-component WebSocket protocol consistency_, conf high
- **Why:** DAY_SETTLEMENT carries the per-month winner, cumulative NAVs, final bonus winner and per-player NAV ranking — the data a strategy needs to know whether it won/lost the month. C++ and JS consumers receive it; Python contestants get nothing, so any callback-driven Python bot is blind to settlement, a cross-SDK functional asymmetry on the same wire message.
- **Trigger:** Run a Python SDK agent through a full month; the server sends DAY_SETTLEMENT at month end; agent.run() never invokes any handler for it (no on_day_settlement exists). Compare to a C++ agent which logs 'Parsed day settlement ...'.
- **Fix:** Add a DaySettlement model + parse_day_settlement in models.py/message.py mirroring DaySettlementMessage (day, month, winnerPlayerId, reason, players[], cumulativeNavs map<int,long>, finalBonusWinnerPlayerId, finalBonusPoints), add an on_day_settlement(self, settlement) hook, and dispatch it on msg_type == 'DAY_SETTLEMENT'.

### 17. Empty / weak JWT_SECRET silently accepted → forgeable admin tokens
- **Where:** `thuai-9-server/submission-api/app/config.py` 4-17 · _eval-submission-api_, conf high

### 18. Data race: run() reads gameState.stage on the main thread while the WebSocket thread reassigns gameState
- **Where:** `sdk-cpp/src/agent.hpp` 104-144, 158-258 · _sdk-cpp_, conf high
- **Why:** Concurrent read of a std::string (gameState.stage) while another thread reassigns the containing struct is a data race = undefined behavior under the C++ memory model. A std::string assignment is not atomic: it can reallocate/free the old buffer and update pointer+size+capacity in several steps. The poll loop can observe a torn string (mismatched pointer/length) or read memory that was just freed by the reassignment, which can yield a garbage stage comparison or a crash. Confirmed: writes at agent.hpp:171/180/188/197/209/237 all occur on the callback thread (ixwebsocket delivers messages on a background thread spawned by ws_.start()), while the read at agent.hpp:136 is on the thread that called run(). The same hazard applies to any user strategy that reads agent.gameState/playerState/marketState from a thread other than the handler callbacks.
- **Trigger:** Run the agent against a live server. Every tick the server broadcasts GAME_STATE, so the WS thread executes `gameState = parseGameState(...)` ~continuously; meanwhile run() reads gameState.stage every 100ms. Under TSan (build with -fsanitize=thread) this fires a data-race report immediately; in production it most often returns a stale/torn value but can fault when the string reassignment reallocates concurrently with the read at end-of-game stage transition.
- **Fix:** Guard the shared snapshot members with a std::mutex (lock in handleMessage around each assignment and in run()/getAllPlayerIds() around each read), or make `stage`/loop-termination signalling an explicit std::atomic flag set inside the callback (e.g. an `std::atomic_bool finished_` set when stage=="Finished") rather than reading a std::string cross-thread. Document that gameState/marketState/playerState are only safe to read from inside the on* handlers (which run on the WS thread).
